# 鰹相場ダッシュボード 憲法・デグレ禁止ルール

この文書は、鰹相場ダッシュボードを継続更新するための基本原則です。個別タスクより優先して確認します。

## 1. データは資産

- 相場データは会社の判断材料であり、仮の数字・推測値・混入データを入れない。
- 価格、数量、船名、日付、港名、サイズ区分は、ひとつでも誤ると判断を歪める前提で扱う。
- 元データと表示データを分けて考える。`market_input.csv` が相場グラフの元、`katsuo_market_data.json` が表示用の生成物。
- 作業で得た知見は `operation_knowledge_base.md` に蓄積し、次回以降の判断を速くする。

## 2. 更新対象の分離

- `data/market_input.csv`: 相場グラフ、最新相場カード、全サイズ相場一覧に使う価格データ。
- `data/katsuo_market_data.json`: CSVから生成される表示用JSON。原則、手編集しない。
- `data/bid_schedule.json`: 入札予定、水揚げ予定、船別明細。相場グラフに入れない情報もここには保持できる。

## 3. 焼津ルール

- 焼津の相場更新は海旋船だけを対象にする。
- 焼津の一本釣り相場は `market_input.csv` に入れない。
- 一本釣り船は、必要に応じて `bid_schedule.json` の入札予定・明細として扱う。
- 海旋船と一本釣り船が同じ日に並ぶ場合、相場グラフに入れるのは海旋船のみ。

## 4. デグレ禁止

- 既存の正しい過去データを、最新データ追加のついでに変更しない。
- 日付・港・サイズ・船名の表記ゆれを増やさない。
- JSON再生成後に、意図しない港・サイズ・船名の消失がないか確認する。
- UIを修正する場合は、相場カード、グラフ、モーダル、入札予定の表示が壊れていないか確認する。
- 自動化スクリプトを修正する場合は、再実行しても同じミスが復活しないようにする。

## 5. 作業前チェック

- 対象の港、日付、船名、漁法を確認したか。
- 相場グラフに入れるデータと、入札予定にだけ残すデータを分けたか。
- 焼津の場合、一本釣りが `market_input.csv` に入っていないか。
- 更新前後で差分を見たか。

## 6. 作業後チェック

- `python scripts/rebuild_data_from_csv.py` を実行したか。
- `python scripts/validate_market_rules.py` を実行したか。
- 対象日付・港の `market_input.csv` と `katsuo_market_data.json` を確認したか。
- `data/bid_schedule.json` に残すべき明細を誤って消していないか。
- 未追跡バックアップやユーザー作成ファイルを触っていないか。

## 7. 判断に迷う場合

- データを推測で補完しない。
- 価格データとして扱うか、入札予定として扱うか迷う場合は、まず `bid_schedule.json` に分離する。
- 焼津の一本釣りを相場グラフに入れる判断はしない。必要な場合はユーザー確認を取る。

## 8. 低トークン運用

- `token_saving_rules.md` に従い、必要なファイルだけを読む。
- ユーザーが「原始人」「短く」「最短」と言った場合は、説明より作業と検証を優先する。
- 最終報告は変更点、検証結果、残作業に絞る。
